# 位置情報探索アプリ

地図上で全国の観光スポットを検索・探索できるWebアプリケーション。
指定した中心点から半径N km以内のスポットを地図とリストで表示し、
地図中心地点の住所をリアルタイムに表示する。

---

## 環境構築

### 前提
- Docker / Docker Compose がインストールされていること
  ※ Node.js・PostgreSQLの個別インストールは不要（すべてコンテナ内で完結）
- Google Maps APIキー（以下2つのAPIを有効化）
  - Maps JavaScript API（地図表示）
  - Geocoding API（逆ジオコーディング＝住所取得）
  ※ APIキーには、使用するAPIのみに制限をかけることを推奨

### セットアップ手順
1. リポジトリをクローン
   ```
   git clone <リポジトリURL>
   cd location-explorer
   ```

2. 環境変数ファイルを作成
   ```
   cp .env.example .env
   ```

3. `.env` にGoogle Maps APIキーを設定
   ```
   # バックエンド（逆ジオコーディング）で使用。ブラウザには露出しない
   GOOGLE_MAPS_API_KEY=＜バックエンド用キー＞

   # フロントエンド（地図表示）で使用。NEXT_PUBLIC_ 付きでブラウザに露出する
   NEXT_PUBLIC_GOOGLE_MAPS_API_KEY=＜フロントエンド用キー＞
   ```
   その他のDB接続設定は、ローカル開発用の初期値が設定済みのため変更不要。

### APIキーの使い分けについて
用途に応じて2つの環境変数でAPIキーを管理している。

- `GOOGLE_MAPS_API_KEY`：バックエンドの逆ジオコーディングで使用。
  サーバー内でのみ使われ、ブラウザには露出しない。
- `NEXT_PUBLIC_GOOGLE_MAPS_API_KEY`：フロントエンドの地図表示で使用。
  Next.jsの仕様上 `NEXT_PUBLIC_` を付けた変数はブラウザに露出するため、
  地図表示（ブラウザ側で動作）に限定して使用し、Google Cloud Console側で
  使用APIの制限・HTTPリファラー制限をかけて保護している。

用途ごとにキーを分離することで、露出範囲を意識した構成としている。

## 実行手順

1. アプリケーションを起動
   ```
   docker compose up
   ```
   ※ 初回起動時は、依存パッケージのインストールとビルドのため数分かかります。

2. 起動時に自動で実行される処理
   DBコンテナの初回起動時、`db/init/` 配下のSQLがファイル名順に自動実行される。
   - `01_init_extension.sql`：PostGIS拡張の有効化
   - `02_create_table.sql`：spotsテーブル作成・GISTインデックス付与
   - `03_import_seed.sql`：シードデータ（CSV約200件）のインポートとlocation生成

   → これにより、初期状態からアプリケーションが動作可能。

3. ブラウザでアクセス
   - フロントエンド： http://localhost:3000
   - バックエンドAPI： http://localhost:3001

### 動作確認（任意）
DBにデータが入っていることを確認する場合：
```
docker compose exec db psql -U postgres -d location_explorer -c "SELECT COUNT(*) FROM spots;"
```

## 使用した主要ライブラリとその選定理由

### 指定技術（課題要件）
以下は課題で指定された技術スタック。
- **Next.js (App Router)**：フロントエンドフレームワーク
- **NestJS**：バックエンドフレームワーク
- **TypeORM**：ORM
- **Tailwind CSS**：CSSフレームワーク
- **PostgreSQL + PostGIS**：データベース（地理空間検索のためPostGISを採用）

### 自身で選定したライブラリ

#### @react-google-maps/api（フロントエンド）
Google MapsをReactで扱うためのライブラリ。生のGoogle Maps
JavaScript APIを直接操作するのではなく、Reactのコンポーネント・
宣言的な記述で地図やマーカーを扱えるため、状態管理との統合が
しやすいと判断し採用した。

#### class-validator / class-transformer（バックエンド）
APIのクエリパラメータ（lat / lng / radius）のバリデーションに使用。
クエリ文字列は文字列で届くため class-transformer で数値に変換した上で、
class-validator で緯度経度の範囲（@IsLatitude / @IsLongitude）や
半径が正の数か（@IsPositive）を検証する。不正な入力をビジネスロジックに
到達させないことで、型安全性とAPIの堅牢性を担保する目的で採用した。

#### pg（バックエンド）
PostgreSQLの接続ドライバ。TypeORMがPostgreSQLと通信するための基盤。
半径検索ではPostGISの `ST_DWithin` をTypeORMのクエリビルダの
where句にSQL式として記述しており、その実行基盤としても利用している。

#### @nestjs/config（バックエンド）
環境変数（DB接続情報・APIキー）を型安全に扱うために採用。
`.env` の値をアプリケーション全体で一元管理する。

## 実装時に特に工夫した点・技術的な判断を行った箇所

### 1. 開発アプローチ：AIエージェントの活用

本課題の実装には、AIエージェント（Claude Code）を全面的に活用した。
実装を丸投げするのではなく、「設計判断は自分で握り、実装を委譲し、
出力を理解・検証する」サイクルを重視した。

- **要件のドキュメント化**：課題要件を自分で分解し、
  要件定義書・技術仕様書・リポジトリ構造定義書に整理した上で
  実装に着手した（例：全国分散データのため初期半径を50kmとする、
  半径検索はPostGISでDB側に委譲する、といった方針を事前に言語化）
- **AIへの前提付与**：定めた設計判断を `CLAUDE.md` にルール化し、
  技術スタックの固定・設計方針・「各実装の理由を説明すること」を
  前提として与え、出力を意図した方針の範囲に保った
- **理解と検証**：生成コードは「なぜその構成・実装か」を都度確認し、
  理解した上で採用した

以降に記載する技術的判断は、いずれもこのプロセスを通じて、
方針決定から検証までを自分で行ったものである。

### 2. 半径検索をDB側（PostGIS）に委譲した

半径N km以内のスポット抽出は、アプリケーション側で全件を取得して
距離計算するのではなく、PostGISの `ST_DWithin` を用いてDB側で
絞り込む設計とした。

- `location` カラムを `GEOGRAPHY(POINT, 4326)` 型で保持し、
  GISTインデックスを付与
- 検索はTypeORMのクエリビルダのwhere句に `ST_DWithin` を
  SQL式として記述して実行
- 座標・半径は文字列に直接埋め込まず、パラメータバインド
  （:lat / :lng / :radiusMeters）でSQLインジェクションを防止

アプリ側で全件距離計算をすると、データ量の増加に比例して処理が
重くなる。DB側でインデックスを効かせて絞り込むことで、
データ量が増えても性能が劣化しにくい構成とした。
（課題目的である「データ設計への配慮」を最も意識した箇所）

### 3. lat/long と location の二重保持

`lat` / `long`（数値）と `location`（PostGIS用のGEOGRAPHY型）を
両方保持している。

- `lat` / `long`：フロントへの返却・マーカー描画に使用
- `location`：半径検索（ST_DWithin）専用の派生カラム

用途が異なるため両持ちとした。マーカー描画のたびにGEOGRAPHY型から
座標を取り出すより、数値で保持する方がシンプルかつ安価なため。

なお `location` は初期化時に `ST_MakePoint(long, lat)` で生成するが、
この関数は第1引数が経度・第2引数が緯度である。取り違えると座標が
大きくずれるため、DB初期化SQLとバックエンドの検索クエリで
引数順序を統一している。

### 4. APIキーの用途別分離

地図表示（フロントエンド）と逆ジオコーディング（バックエンド）で
APIキーを分離した。

- フロントエンド用：Maps JavaScript API に制限 + HTTPリファラー制限
  （`NEXT_PUBLIC_` によりブラウザに露出するため、露出前提で防御）
- バックエンド用：Geocoding API に制限（サーバー内で使用し露出しない）

用途ごとにキーと権限を分けることで、一方が漏洩しても
影響範囲を限定できる構成とした。

### 5. 逆ジオコーディングをバックエンド経由にし、キャッシュを挟んだ

地図中心の住所取得（逆ジオコーディング）は、フロントから直接
Google APIを呼ばず、バックエンド経由とした。

- APIキーをブラウザに露出させずに済む
- バックエンド側でキャッシュを挟み、Google APIへの
  再問い合わせ（課金・レート制限対象）を削減できる

キャッシュのキーは緯度経度を小数点3桁（約100m格子）に丸めて
生成し、わずかに異なる座標でも同一の住所結果を再利用できる
ようにした。住所は時間で変化しないためTTLは設けていない。

### 6. 住所取得におけるAPI呼び出しの多層的な抑制

地図移動時の逆ジオコーディングは、3段階で無駄な呼び出しを抑えている。

1. FE側の移動距離判定：前回取得地点からほぼ動いていなければ
   リクエスト自体を発生させない
2. デバウンス：移動が止まってから500ms後に呼び出す
   （住所表示は要件上「リアルタイム・高頻度」が求められるため、
   追従性を優先しつつ呼び出しを抑える値として設定）
3. 移動中の中断：待機中に再移動があれば、AbortControllerで
   進行中のリクエストを中断する（古い結果が最新表示を上書きする
   競合を防ぐ）

FE側の間引きとBE側のキャッシュを併用することで、FE→BEの通信と
Google APIへの問い合わせの両方を削減している。

### 7. 逆ジオコーディング結果の整形とエッジケース処理

住所表示は、取得結果を3つのケースに分けて処理している。

- 国内の住所：Google APIの `formatted_address` から国名（日本）と
  郵便番号を除去し、「今どこを見ているか」が把握しやすい表示に整形
- 日本国外：address_componentsの国コード（JP以外）で判定し
  「日本国外」と表示。スポットは国内データのみのため、
  海外の詳細住所やPlus Codeを表示しない
- 住所がない地点（海上等）：ZERO_RESULTSをnullとして扱い、
  エラーにせず欠損として処理

外部API依存の失敗（接続エラー・HTTPエラー・APIステータス異常）も
それぞれログ出力の上で例外として処理している。

### 8. 入力バリデーション（DTO）

APIのクエリ（lat / lng / radius）は、NestJSのDTOと
class-validatorで検証している。クエリ文字列は文字列で届くため
数値に変換した上で、緯度経度は妥当な範囲か（@IsLatitude /
@IsLongitude）、半径は正の数か（@IsPositive）を、
サービス層に到達する前にチェックする。

### 9. 起動順序の保証

- DB初期化SQLは連番（01拡張有効化 → 02テーブル作成 →
  03インポート）で実行順を保証
- バックエンドは `depends_on: condition: service_healthy` で
  DBの初期化完了（pg_isready）を待ってから起動

初期化途中のDBに接続してテーブル未作成エラーになることを防ぐ。

### 10. 中心地点マーカーの表示

地図中心の住所テキストに加え、中心点を示すマーカーを表示した。
地図上で中心点が視覚的に分かりづらく、類似の地図アプリでも
中心をマーカーで示す実装が一般的なため、
「今どこを見ているか」の把握を助けると判断した。
スポットのマーカーとは色・デザインを分けて区別している。

### 11. リストのカテゴリ内を距離順に表示

スポット一覧は、BEから返却された順（PostGISの`ST_DWithin`による絞り込み順で、
距離順は保証されない）をそのまま表示するのではなく、カテゴリ内で中心点からの
距離が近い順に並べ替えた上で、各スポットに距離（km）を併記している。

距離計算には、逆ジオコーディングの移動距離判定で既に使っている
`frontend/src/lib/geo.ts`の`distanceMeters`（Haversine公式）を再利用した。
用途は異なる（一方は「わずかな移動」の判定、一方は「近い順」の並べ替え）が、
どちらも同じ「2点間の実距離をメートルで求める」計算であるため、
ロジックを分離せず共通化した。

## 時間が足りず簡略化した箇所・今後の改善点

### テストコードの整備
今回はテストコードを実装できていない。本来は、半径検索
（ST_DWithinによる絞り込み）のロジックや、住所整形・国内外判定などの
ユニットテストを整備すべきである。特に半径検索とキャッシュのキー丸めは、
境界値でのテストを用意することで安全にリファクタリングできる状態にしたい。

### 現在地を起点とした初期表示
初期表示は東京駅に固定している。ユーザーの現在地（Geolocation API）を
起点にする案も検討したが、本アプリのスポットは全国の観光地で分散して
いるため、ユーザーの現在地周辺が0件になり、起動直後に機能が伝わらない
リスクがあると判断し見送った。改善するなら、現在地を取得しつつ周辺に
スポットがない場合は検索範囲を自動で広げる、あるいは「現在地へ移動」
ボタンを設けて初期表示は東京駅のままにする、といった折衷案が考えられる。

### APIキーのアクセス制限の強化
地図表示用とバックエンド用でAPIキーは分離し、それぞれ使用APIを
制限した上で、フロント用にはHTTPリファラー制限をかけている。一方、
バックエンド用キーはローカル開発の都合上IP制限をかけていない。
本番運用では、バックエンドの固定IPやサーバー環境に応じたIP制限を
かけることで、さらに露出範囲を狭められる。

### キャッシュの永続化・共有
逆ジオコーディングのキャッシュは、プロセス内のインメモリ実装（Map）と
している。シンプルで高速だが、プロセス再起動で消え、複数インスタンスに
スケールした場合は共有されない。本番でキャッシュの効果を最大化するなら、
Redis等の外部キャッシュに置き換え、インスタンス間で共有・永続化する
構成が望ましい。

### レスポンシブ対応
今回はPC画面での表示を前提に実装しており、メディアクエリによる
スマートフォン・タブレット向けのレイアウト最適化は行っていない。
地図とスポット一覧を横並びにしている現状のレイアウトは、狭い画面では
見づらくなるため、改善するなら画面幅に応じて地図とリストを縦積みに
切り替える、リストをドロワー化するなどのレスポンシブ対応が考えられる。

### スポットの管理機能
本アプリは探索（検索・表示）に特化しており、スポットの登録・編集・削除
（CRUD）は対象外としている。シードデータからの読み取り専用の想定のため
今回は実装していないが、運用を想定するなら管理機能の追加が考えられる。
